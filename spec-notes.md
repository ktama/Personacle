# spec-notes: Personacle 要件定義 作業メモ

## Phase 0: タスク理解

依頼の言い直し:
SLM(小規模言語モデル)をベースに、永続的な記憶領域を持つ「人格を持ったAIエージェント」を作る。
エージェントはユーザーと1対1で対話できるだけでなく、複数のAIエージェント同士でも対話できる。
まず仕様(要件定義)を固めてから設計・実装に進む。

### 既知

- ベースはSLM(大規模モデルではなく小規模モデルを使う意図がある)
- 記憶領域を持たせる(会話をまたいで何かを保持する)
- 「人格」を持たせる(キャラクター性・一貫した振る舞い)
- 複数AI間の対話機能が必要
- ユーザーとAIの対話機能が必要
- リポジトリ名は Personacle(新規・空リポジトリ)
- 開発環境は Windows 11

### 未知

1. SLMの実行方式: ローカル実行(Ollama / llama.cpp 等)か、クラウドAPIか。SLM指定の動機(コスト? プライバシー? オフライン? 研究?)
2. 利用形態: 個人がローカルで使うツールか、複数ユーザー向けサービスか
3. UI: CLI / デスクトップ / Web のどれか
4. 複数AI対話の形態: AI同士が自律的に会話するのを観察する型か、ユーザーも参加するグループチャット型か
5. 記憶の期待内容: 会話履歴の要約? ユーザーに関する事実? AI自身の経験・感情?
6. 人格の可変性: 人格は作成時に固定か、対話経験によって記憶・人格が変化(成長)するか
7. 人格の作成方法: ユーザーが自由に定義するか、プリセットか
8. 同時に存在するAIの数の想定(2〜3体か、数十体か)
9. 利用目的: 趣味/エンタメ、研究、実用アシスタント

### 前提(未確認の仮定)

- シングルユーザー(開発者本人)がローカルで使う個人プロジェクトである
- 日本語での対話が主である
- リアルタイム音声は不要でテキスト対話である
- 商用サービス水準の認証・課金・スケーラビリティは不要である

## Phase 1: 質問と回答

1. 実行環境 → **ローカル実行**(Ollama / llama.cpp 系)
2. AI間対話の形態 → **自律会話の観察 + 1対1個別対話が主**(グループチャットは対象外)
3. 人格・記憶の成長 → **記憶も人格も成長する**
4. 目的 → **将来公開したいプロダクト**
5. 優先度の追加確認 → 人格成長・自律会話とも **Must**

## v0.2 追補の要件定義 作業メモ (2026-07-08 開始)

### Phase 0 (v0.2): タスク理解

言い直し:
v0.1.0(FR-01〜19全実装済み)のPersonacleに、次期バージョンとして
テーマA(育成実感: 時間経過認識・記憶整理統合・ムード・日記)、
テーマB(可視化: 成長ダッシュボード・関係図・記憶検索)、
テーマC(先送り分: グループチャット・自律会話品質向上)の9機能を追加する。
成果物は既存要件書 docs/requirements.md v1.1 への追補(v2.0)。

#### 既知 (v0.2)

- v0.1.0の全FR/NFR/EC実装済み。要件書v1.1・設計書v1.4とも承認済み
- 既存資産: 性格5軸+親密度+印象メモ、記憶(kind 4種・importance・埋め込み)、
  personality_event(変化履歴の追記ログ)、セッション後処理(非同期バッチ)、
  自律会話2〜6体(ラウンドロビン)、記憶上限はアーカイブ方式(EC-06)
- 制約継続: ローカル完結・シングルユーザー・Windows 11・日本語のみ・SLM
- ダッシュボード/関係図の素材データ(変化履歴・関係性)は既に永続化されている
- グループチャットはv1.1 non-goals筆頭。自律会話の停滞・破綻検出はEC-11/12で先送り済み
- ユーザーはテーマA・B・Cの9機能すべてへの対応を決定済み

#### 未知 (v0.2)

1. 時間経過認識の範囲: 応答への反映のみか、ペルソナからの自発的な話しかけまで含むか。OS通知まで踏むか
2. 記憶整理・統合の自動性: 完全自動か、ユーザー承認制か。統合後の元記憶の扱い(アーカイブ/削除/保持)
3. ムードの持続範囲と可視性: セッション内限りか持続するか。ユーザーに表示するか隠すか
4. 日記の生成単位: セッションごとか日ごとか。日記は想起に使うか読み物専用か
5. グループチャットの応答方式: ユーザー発話後に誰が・何体応答するか。指名の可否。参加上限
6. 自律会話品質向上の範囲: 停滞検出のみか人格破綻検出も含むか。検出時の動作(終了/話題転換/表示のみ)
7. 9機能のMoSCoW配分とリリース単位(一括v0.2か分割か)
8. 追補の文書形態: requirements.md本体の改版か、別ファイルか

#### 前提(仮定) (v0.2)

- (仮定) v1.1の前提・制約(ローカル完結、Windows 11、日本語、メモリ16GB・GPUなし)は変更しない
- (仮定) 新機能の推論はすべて既存接続仕様(chat completions / embeddings)の範囲で行い、新たな外部依存を増やさない
- (仮定) NFR-01〜07は新機能にもそのまま適用される(特に応答経路に推論を追加しない方針)
- (仮定) 追補は requirements.md を v2.0 に改版する形とする(単一の正を維持)。FR-20〜の連番

### Phase 1 (v0.2): 質問と回答 (2026-07-08)

1. 自発性の範囲 → **アプリ内の話しかけまで**(応答への時間経過反映+チャット画面を開いたときのペルソナからの最初の一言。OS通知は対象外)
2. 記憶統合の自動性 → **自動+元記憶はアーカイブ**(想起対象外・閲覧可。誤統合はFR-11の編集で是正)
3. グループチャットの応答方式 → **自動選択1体+指名可**(全員応答はSLM速度でテンポが崩れるため不採用)
4. 自律会話の品質向上 → **停滞検出+話題転換**(人格破綻の自動検出は次版以降)

残る未知3,4(ムードの持続・可視性、日記の単位・用途)は仮定を置いてPhase 2で合意を取る:

- (仮定) ムードはセッションをまたいで持続し、時間経過で平常へ回帰する。現在のムードはユーザーに表示する
- (仮定) 日記は「対話があった日」ごとに1件生成する読み物であり、想起には使わない(想起は既存の記憶で担う。二重管理を避ける)

### Phase 2 (v0.2): スコープ案

目的: v0.1で仕組みとして実装した記憶と成長を、ユーザーが「体感できる」ものにする。あわせて初版で先送りした対話形態(グループチャット)と自律会話の品質を拡張する。

MoSCoW案:

- Must: 時間経過認識と話しかけ(A-1) / 記憶の整理・統合(A-2) / 記憶検索(B-7) / グループチャット(C-8)
- Should: ムード(A-3) / 日記(A-4) / 成長ダッシュボード(B-5) / 停滞検出+話題転換(C-9)
- Could: 関係図(B-6)
- Won't(v0.2): OS通知・常駐、人格破綻の自動検出、記憶統合の承認フロー、v1.1由来のnon-goals(音声・クラウド・多言語・他OS)
